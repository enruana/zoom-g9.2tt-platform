## Story Information

**Story Type:** Hybrid Origin Story + Process Story

**Framework Used:** Origin Story (columna vertebral emocional) + Process Story (carne técnica)

**Purpose:** Presentación de 1 hora para dev meetup regional. Slides con pseudo-código, diagramas estilo dibujo (animados si posible). Mostrar cómo se construyó la plataforma Zoom G9.2tt usando IA (Claude Code) — enfocado en el CÓMO, no solo el QUÉ.

**Target Audience:** Comunidad de desarrolladores regional — mezcla de niveles (juniors a seniors), diversas especialidades. No necesariamente usuarios de IA activos.

**Key Messages:**
1. La IA no reemplaza al desarrollador, potencia increíblemente si ya tienes experiencia y criterio
2. El proceso cognitivo humano no se acelera — el aprendizaje sigue siendo humano. Los resultados sí se multiplican 10x
3. Ya no necesitas memorizar detalles técnicos, necesitas entender la intencionalidad detrás de las decisiones
4. Seguir aprendiendo como siempre: práctica + teoría. Pero amplificar cada paso de experimentación con IA
5. En cada experimento, usar IA para descubrir qué más se puede aplicar y mejorar basado en el contexto puntual
6. Mostrar complejidad técnica real (reverse engineering, MIDI SysEx, CRC-32, React architecture) pero demostrar que no necesitas dominar cada detalle profundo para ejecutar

**Constraints:**
- Duración: 1 hora
- Formato: Slides, pseudo-código, diagramas tipo dibujo (animados si posible)
- Tono: Sobrio, sin magia, honesto. Línea narrativa clara
- Profundidad técnica: Suficiente para apreciar la complejidad, no tanto como para perderse en ella

**Title:** "Lo que se puede medir, se puede controlar"

## ACTO I — "Lo que se puede medir" (15 minutos)

### Slide 5 — El Muro
> [VISUAL: Diagrama del protocolo MIDI — bloques de bytes hexadecimales, flechas entre pedal y computador. Se ve intimidante a propósito.]

**Lo que dices:**

"El primer obstáculo fue el protocolo. El G9.2tt se comunica por MIDI — pero no MIDI estándar. Es un protocolo propietario de Zoom. SysEx — System Exclusive messages. Cerrado. Sin documentación pública.

Yo nunca había trabajado con MIDI. No sabía cómo estaba estructurado, cómo se leían los patches, cómo se escribían. No tenía absolutamente ninguna idea de cómo funcionaba esta comunicación.

Y al ver que el protocolo era cerrado, pensé: esto va a ser más difícil de lo que creí."

---

### Slide 6 — El Principio
> [VISUAL: Fondo limpio. Una sola frase grande, centrada: "Lo que se puede medir, se puede controlar."]

**Lo que dices:**

"Pero hay algo que aprendí hace mucho tiempo y que siempre me ha servido:

*Lo que se puede medir, se puede controlar.*

Si puedo capturar el tráfico entre el pedal y la computadora... puedo guardarlo. Si puedo guardarlo, puedo analizarlo. Si puedo analizarlo, puedo entenderlo. Y si lo entiendo, puedo replicarlo.

Ese fue el primer paso. No necesitaba entender MIDI. Necesitaba poder OBSERVARLO."

---

### Slide 7 — La Captura
> [VISUAL: Diagrama sketch animado: Pedal G9.2tt ↔ Cable MIDI DIN ↔ Interfaz USB-MIDI ↔ Raspberry Pi capturando tráfico ↔ Archivos de datos]

**Lo que dices:**

"Monté un sistema de captura. El pedal se comunica por cables MIDI DIN — los clásicos de 5 pines. Conecté una interfaz USB-MIDI, escribí scripts en Python para interceptar todo el tráfico, y empecé a grabar sesiones completas.

Cada vez que el software original leía un patch, yo capturaba los bytes. Cada vez que escribía, los grababa. Acumulé datos."

---

### Slide 8 — Los Datos Crudos
> [VISUAL: Pantalla de terminal con datos hexadecimales reales — F0 52 00 42 21... fragmentos del protocolo real. No se espera que la audiencia los lea.]

**Lo que dices:**

"Esto es lo que se ve. Bytes crudos. SysEx messages. F0 es inicio, F7 es fin, 52 es Zoom, 42 es el modelo G9.2tt.

No les voy a pedir que entiendan esto. Lo muestro para que vean la complejidad real. Esto es ingeniería inversa: datos sin contexto, sin manual, sin guía. Solo patrones esperando ser descubiertos.

Y aquí es donde empieza la parte interesante."

---

### Slide 9 — El Análisis con IA
> [VISUAL: Split screen sketch — Izquierda: datos hexadecimales. Derecha: Claude Code analizando, con flechas señalando patrones descubiertos. Texto: "Nibble encoding", "7-bit encoding", "128 bytes por patch"]

**Lo que dices:**

"Aquí entró Claude Code por primera vez. Le di los datos capturados y le dije: necesito entender este protocolo.

Y empezamos a trabajar juntos. Yo le daba contexto — 'esto parece un header', 'estos bytes cambian cuando modifico el volumen' — y la IA analizaba patrones a una velocidad que yo no podría igualar.

Descubrimos que los datos del patch se codifican en dos capas: primero nibble encoding — cada byte se parte en dos mitades — y después 7-bit encoding para que sea compatible con MIDI. 128 bytes de datos reales se convierten en 147 bytes transmitidos.

¿Necesité memorizarme los detalles de nibble encoding? No. Necesité entender QUÉ estaba pasando y POR QUÉ. La IA manejó el CÓMO."

### Slide 11 — Pero Faltaba Algo
> [VISUAL: Diagrama de un patch: datos + ??? (un bloque con signo de interrogación). Texto: "El Checksum"]

**Lo que dices:**

"Pero había un problema. Para ESCRIBIR patches al pedal — no solo leerlos — necesitábamos un checksum. Un valor de verificación que el pedal calcula para asegurarse de que los datos no están corruptos.

Sin el checksum correcto, el pedal rechaza la escritura. Silenciosamente. Sin error. Simplemente la ignora."

## ACTO II — "El Momento que Cambió Todo" (15 minutos)

### Slide 12 — El Muro del Checksum
> [VISUAL: Diagrama: Datos capturados → flechas hacia múltiples algoritmos (CRC-16, CRC-32, XOR, suma simple) → todos con ❌]

**Lo que dices:**

"Intentamos de todo. Tomamos los datos capturados — pares de patch + checksum — y probamos cada algoritmo conocido. CRC-16, CRC-32 con distintos polinomios, XOR, sumas simples, checksums de Fletcher.

Nada coincidía.

El problema es que sin saber sobre QUÉ datos se calcula el checksum — si es sobre los datos crudos, los codificados, con header, sin header — las combinaciones son enormes. Hubiera sido prácticamente imposible encontrarlo así."

---

### Slide 13 — La Sugerencia Inesperada
> [VISUAL: Chat simplificado — Claude Code: "El G9ED está construido en .NET. Podemos hacer decompilación del ejecutable para encontrar el algoritmo directamente en el código fuente." Una flecha grande apuntando a un archivo .exe]

**Lo que dices:**

"Y aquí pasó algo que no se me hubiera ocurrido.

Yo le estaba explicando a Claude Code el contexto del problema. Le mencioné que el software original era un .exe construido en .NET. Y me dijo algo que me detuvo en seco:

'Si es .NET, podemos hacer decompilación. Podemos recuperar el código fuente del ejecutable y buscar el algoritmo del checksum directamente ahí.'

*(pausa)*

Yo no sabía que eso era posible. Nunca lo había pensado. Pero la IA sí conocía esa técnica. Y solo pudo sugerirla porque YO le di el contexto: que era .NET, que necesitábamos un checksum, que no lo encontrábamos en los datos."

---

Ni yo solo, ni la IA sola, hubiéramos llegado ahí. Fue la combinación."

---

### Slide 15 — El CRC-32
> [VISUAL: Pseudo-código limpio del algoritmo CRC-32:
> ```
> crc = 0xFFFFFFFF
> for byte in patch_data[0..128]:
>     crc = crc ^ byte
>     for bit in 0..8:
>         if crc & 1:
>             crc = (crc >> 1) ^ 0xEDB88320
>         else:
>             crc = crc >> 1
> checksum = crc ^ 0xFFFFFFFF
> ```
> Texto: "Polinomio estándar. Calculado sobre los 128 bytes crudos."]

**Lo que dices:**

"Lo encontramos. CRC-32, polinomio estándar, calculado sobre los 128 bytes crudos del patch — antes de cualquier encoding.

No necesito que memoricen este código. Lo muestro para que vean que es un algoritmo real, específico, que estaba escondido dentro de un ejecutable de 2007. Y que una combinación de contexto humano y capacidad de la IA lo sacó a la luz."

---

### Slide 16 — El Momento Eureka

*(pausa)*

## Story Elements Analysis

### Character/Voice
El presentador habla desde la experiencia personal, sin arrogancia. Tono conversacional pero estructurado. Usa "yo" para las anécdotas y "ustedes" para las invitaciones. No predica — comparte.

### Conflict/Tension
- Pedal hermoso pero software muerto
- Protocolo cerrado sin documentación
- Checksum imposible de descifrar por fuerza bruta
- IA que rompe lo que ya funciona
- La pregunta implícita: ¿vale la pena aprender si la IA ya sabe?

### Transformation/Change
- De frustración ("fue mala idea comprar esto") a creación ("construí algo que no existía")
- De "la IA lo hace todo" a "la IA amplifica lo que yo sé"
- De audiencia pasiva a audiencia con ganas de empezar algo

### Emotional Touchpoints
1. La risa incómoda del Raspberry Pi (todos hemos hecho algo así)
2. El respeto por la complejidad del protocolo (datos hex reales)
3. El asombro del decompile (la IA sugirió algo inesperado)
4. El momento eureka del patch cambiando en el pedal
5. La honestidad de "la IA rompe cosas"
6. La invitación final ("¿Qué van a construir?")

### Key Messages
1. Lo que se puede medir, se puede controlar
2. La IA potencia, no reemplaza
3. Tu proceso cognitivo no se acelera — tus resultados sí
4. Ya no memorices detalles — entiende intenciones
5. Sigue aprendiendo. Amplifica con IA. Empieza hoy.
